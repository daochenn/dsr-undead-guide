# DS3 Stat Sweeper — план таски

Инструмент-исследователь для выяснения, **почему за правку сейва через сейв-эдитор дают бан**, а за
Cheat Engine (`DS3_TGA_v3.4.0.CT`) — нет.

Идея: воспроизвести ровно то, что делает CT (правит характеристики **в памяти процесса**), прогнать
выбранную характеристику от 1 до 99, после каждого шага заставить игру сохраниться (ESC×2), и собрать
99 сейвов. Потом эти 99 «чистых» (сделанных игрой) сейвов сравниваются с тем, что производит наш
сейв-эдитор, — так найдём отличие, из-за которого прилетает бан.

- **Язык:** Python + `pymem`
- **Что пишем в .bin:** расшифрованный слот выбранного персонажа (не весь `.sl2`)
- **Сейв:** авто-фокус окна DS3 + `SendInput` scancode ESC×2, ждём смену mtime у `.sl2`
- **Верификация:** парсим `.sl2` после сейва (BND4 + AES-CBC) и читаем стат из файла

---

## 1. Вход (аргументы запуска)

| Аргумент | Пример | Назначение |
|---|---|---|
| `--steam-id` | `011000012ff75f05` | Папка сейва: `%APPDATA%\DarkSoulsIII\<steam-id>\DS30000.sl2` |
| `--slot` | `0` | Номер персонажа (BND4-entry 0..9), какой слот верифицируем/выгружаем |
| `--stat` | `VIT` | Характеристика для прогона: VIG/ATN/END/VIT/STR/DEX/INT/FTH/LCK |
| `--out` | `./out/VIT` | Куда складывать `VIT_01.bin ... VIT_99.bin` |
| `--from`/`--to` | `1` / `99` | Диапазон (по умолчанию 1..99) |

На вход дают Steam ID (выбор папки сейва) и номер персонажа. Активная папка на этой машине сейчас —
`011000012ff75f05` (её `DS30000.sl2` изменялся сегодня).

> ⚠️ Слот в памяти (в игре загружен ровно 1 персонаж) и слот в файле (`--slot`, BND4-entry) — это
> разные вещи. В памяти правим того персонажа, что **сейчас загружен в игре**; в файле верифицируем
> `--slot`. Оператор должен зайти в игру именно нужным персонажем и указать его же номер в `--slot`.

## 2. Модуль памяти (порт из CT) — `memory.py`

### 2.1 Базовый указатель WorldChrMan
CT берёт его AOB-сканом модуля `DarkSoulsIII.exe`, дизассемблирует найденную инструкцию
`mov reg,[rip+disp32]` и берёт абсолютный операнд `[addr]` как **адрес переменной-указателя**
(`registerBaseByKey`, строки 504–525 в .CT). Затем в чит-энтри `[WorldChrMan]` — это разыменование.

Порт на Python:
1. Прочитать образ `DarkSoulsIII.exe`, найти AOB (кандидаты):
   - основной (актуальный): `48 8B 05 ?? ?? ?? ?? 48 85 C0 74 0F 48 39 88`
   - из .CT (BaseB, закомментирован): `48 8B 1D ?? ?? ?? 04 48 8B F9 48 85 DB ...`
2. От адреса инструкции разобрать RIP-relative: `ptr_var = instr_addr + instr_len + disp32`
   (для `48 8B 05 dd dd dd dd`: `instr_len = 7`, `disp32` — байты 3..6).
3. `world_chr_man = read_u64(ptr_var)` — это и есть `[WorldChrMan]`.

> Проверка корректности AOB: после резолва прочитать Level по цепочке (ниже) — должен совпасть с
> тем, что в игре. Если AOB не тот (обновление игры) — свериться с адресом WorldChrMan в самом CT.

### 2.2 Цепочка указателей до атрибутов
Из группы **`--- Attributes`** в CT (это именно та группа, что нужна). CE читает офсеты снизу вверх,
т.е. список `[off, 1FA0, 38, 40]` = `[[[[WorldChrMan]+40]+38]+1FA0]+off`.

```
p = read_u64(world_chr_man + 0x40)
p = read_u64(p + 0x38)
p = read_u64(p + 0x1FA0)
stat_addr = p + STAT_OFFSET      # 4 байта, little-endian
```

**Офсеты атрибутов (4 байта каждый), из CT (Player 1 / Attributes):**

| Стат | Ключ | Офсет |
|---|---|---|
| Vigor | VIG | `0x44` |
| Attunement | ATN | `0x48` |
| Endurance | END | `0x4C` |
| Strength | STR | `0x50` |
| Dexterity | DEX | `0x54` |
| Intelligence | INT | `0x58` |
| Faith | FTH | `0x5C` |
| Luck | LCK | `0x60` |
| Vitality | VIT | `0x6C` |
| (справка) Level | — | `0x70` |

### 2.3 API модуля
- `attach() -> Pymem` (по имени процесса `DarkSoulsIII.exe`)
- `resolve_worldchrman() -> int`
- `resolve_stat_addr(stat_key) -> int`
- `read_stat(stat_key) -> int` / `write_stat(stat_key, value)` (RPM/WPM 4 байта LE)

> ⚠️ Пишем **только сам стат** — как это делает CT. Level/Souls/HP не трогаем. Именно расхождение
> «стат поднят, а level/soul-memory не пересчитаны» — один из главных подозреваемых на бан, и его надо
> увидеть в сравнении сейвов, а не «исправить».

## 3. Модуль автосейва (ESC×2) — `input_save.py`

Игра сохраняется при открытии/закрытии меню. Последовательность: ESC (открыть), ~1 c, ESC (закрыть).

1. **Фокус окна:** найти окно с заголовком `DARK SOULS III` (`EnumWindows` + `GetWindowText`),
   `SetForegroundWindow` / `ShowWindow(SW_RESTORE)`. Дать ~0.3 c на активацию.
2. **ESC scancode через `SendInput`:** DS3 слушает DirectInput/raw input, поэтому шлём **scancode**,
   а не virtual-key: `KEYBDINPUT` c флагом `KEYEVENTF_SCANCODE`, scancode ESC = `0x01`.
   - down (`0x01`) → up (`0x01 | KEYEVENTF_KEYUP`) — один «нажал-отпустил»
   - пауза ~1 c
   - ещё один «нажал-отпустил»
3. **Дождаться нового сейва:** запомнить `mtime`+размер `DS30000.sl2` **до** ESC, после — поллить,
   пока `mtime` не изменится (таймаут ~15 c). Смена mtime = игра записала сейв.

> Тонкости: раскладка/оконный режим не влияют на scancode. Если игра в эксклюзивном фуллскрине,
> `SetForegroundWindow` может не сработать — на время прогона поставить **Borderless/Windowed** в
> настройках DS3. Оператор в этот момент не трогает мышь/клавиатуру.

## 4. Модуль верификации/выгрузки `.sl2` — `sl2.py`

Формат уже разобран в проекте: `src/apps/ds3/lib/SaveFileEditor.ts`, `Character.ts`, `constants.ts`.
Портируем на Python (ключ и офсеты берём оттуда).

### 4.1 Разбор BND4
- Заголовок BND4 `0x40` байт; сигнатура `"BND4"`.
- Число энтри: `u32` @ `0x0C`.
- Заголовок энтри `i`: `0x40 + i*0x20`; размер энтри `u64` @ `+0x08`; data offset `u32` @ `+0x10`.
- Тело энтри: `[MD5 16][IV 16][AES-CBC зашифрованные данные]`.
- Расшифровка: **AES-128-CBC**, ключ (`constants.ts` `AES_KEY`):
  `FD 46 4D 69 5E 69 A3 9A 10 E3 19 A7 AC E8 B7 FA`, IV — 16 байт из тела энтри.
- (опц.) проверить MD5 от `IV+encrypted` против сохранённого чек-суммы.

### 4.2 Чтение стата из расшифрованного слота
- Найти паттерн персонажа (32 байта, `FF FF FF FF 00×12` дважды) — `CHARACTER_PATTERN`.
- Стат = `pattern_off + RELATIVE_OFFSET`. Из `constants.ts` (относительно паттерна):
  VIG `-0x10C`, ATN `-0x108`, END `-0x104`, VIT `-0xE4`, STR `-0x100`, DEX `-0xFC`, INT `-0xF8`,
  FTH `-0xF4`, LCK `-0xF0`, LEVEL `-0xE0`.

### 4.3 API
- `decrypt_slot(sl2_bytes, slot_index) -> bytes` (расшифрованный слот — это и есть `.bin`)
- `read_stat_from_slot(slot_bytes, stat_key) -> int`

## 5. Главный сценарий — `main.py`

```
attach(process)
wcm = resolve_worldchrman()          # + sanity-check по Level
for value in range(from, to+1):      # 1..99
    write_stat(stat, value)          # как CT — только сам стат
    assert read_stat(stat) == value  # записалось в память
    mtime0 = stat_of(sl2).mtime
    focus_ds3_window()
    press_esc(); sleep(1); press_esc()
    wait_until_mtime_changed(sl2, mtime0, timeout=15)
    sl2_bytes = read_file(sl2)
    slot = decrypt_slot(sl2_bytes, args.slot)
    got = read_stat_from_slot(slot, stat)
    assert got == value              # в ФАЙЛЕ актуальное значение
    write_file(out / f"{stat}_{value:02d}.bin", slot)
    log(value, got, ok=(got==value))
```

Результат: `out/VIT/VIT_01.bin ... VIT_99.bin` — 99 расшифрованных слотов персонажа `--slot`,
каждый соответствует Vit = N, сделанный самой игрой.

## 7. Риски и гипотезы про бан (ради чего всё)

1. **Главный риск №1 — порт кода из CT.** Резолв WorldChrMan (AOB + RIP-relative) и цепочка
   `+40/+38/+1FA0/+off`. Проверяется sanity-чеком: Level из памяти == Level в игре.
2. **Главный риск №2 — ESC×2 доходит до игры.** Нужен scancode (не VK) + фокус окна + Borderless.
   Проверяется по смене mtime `.sl2`.
3. **EAC / онлайн.** DS3 использует Easy Anti-Cheat. Чтение/запись памяти и «странные» сейвы
   безопаснее проверять в **оффлайн-режиме** (Steam offline / игра без сети). Вероятно, CT «не банят»
   именно потому, что им пользуются оффлайн. Прогон делать оффлайн.
4. **Собственно гипотеза бана.** Наш сейв-эдитор, скорее всего, пишет стат, но **не пересчитывает
   связанные поля так, как это делает игра** (level, soul memory, производные HP/FP/Stamina,
   контрольные суммы/структуры вне слота). 99 «эталонных» сейвов от игры позволят сделать бинарный
   diff с выводом нашего эдитора и точно локализовать расхождение.

# РЕЗУЛЬТАТЫ ИССЛЕДОВАНИЯ (обновлено 2026-07-12)

## Реализация (что по факту работает, отличия от плана выше)

- **База памяти — НЕ WorldChrMan, а GameDataMan.** Цепочка Attributes из CT (`[[[[WorldChrMan]+40]+38]+1FA0]`)
  — это путь **онлайн-сессии**, пуст в оффлайне (`+0x40` → 1). Рабочий оффлайн-путь:
  **`[[GameDataMan]+0x10]` = PlayerGameData**, AOB `48 8B 05 ?? ?? ?? ?? 48 85 C0 ?? ?? 48 8B 40 ?? C3`.
  Офсеты статов те же (VIG 0x44 … VIT 0x6C, Level 0x70).
- **Level пишем сами** = сумма 9 атрибутов − 89 (`write_stat` в `memory.py`).
- **ESC-сейв — полуавтомат**: программный фокус DS3 не принимает. Отсчёт 5 c (юзер разворачивает игру),
  затем ESC×2 (scancode 0x01, `KEYEVENTF_SCANCODE`) с паузой 0.2 c; при неизменившемся mtime — до 3 повторов.
- **`main.py`**: за один запуск гоняет все статы (цикл в цикле), перед каждым выставляет остальные 8 в baseline=11.
- **Доп. инструменты**: `make_chars.py` (создать 10 персонажей в слотах), `diff_chars.py` / `diff_derived.py`
  (сравнение before/after), `sl2.py` (BND4+AES, чтение стата из файла).

## Карта меняющихся байт слота (относительно паттерна персонажа)

Из свипа (память+ресейв игрой) меняются всего **49 байт**. Классификация:
- 9 статов (`−0x10C … −0xE4`, каждый `== value`), **Level** (`−0xE0`, `== level`).
- **HP**: `−0x130` = max HP (по `VIGOR_TO_HP`), `−0x134` = HP-A (= current HP с угольком, 1.3× max). Только Vigor.
- **FP**: `−0x124` = max FP, `−0x128` = FP-A (всегда == max). Только Attunement.
- **Stamina**: `−0x11c`, `−0x118`, `−0x114` — **три ИДЕНТИЧНЫЕ копии** (всегда равны), только Endurance,
  `== ENDURANCE_TO_STAMINA[END]`.
- **Резисты** (5 полей `−0x70 … −0x60`): база 107 + f(level) + доп. от конкретного стата
  (`−0x60`←VIG, `−0x6c`←END, `−0x70`/`−0x68`←VIT, `−0x64`←LCK). Похоже Poison/Bleed/Frost/Curse.
- **Служебная шапка слота** (в начале, не у паттерна): `0x0C` = playtime/счётчик (монотонно растёт),
  `0x24` (4 б) и `0x60` (16 б) = внутренние хеши/nonce — **из данных не реверсятся** (ключ/соль),
  все 891 значения `0x60` уникальны.

## ГЛАВНЫЙ ВЫВОД — причина бана (эксперимент с 10 персонажами)

Создали 10 персонажей редактором (base11 + 9× один-стат-99), зашли каждым в игру, сравнили before/after.
**Игра при загрузке сейва НЕ пересчитывает базовые HP / FP / Stamina — доверяет тому, что в файле.**
Пример: слот VIG99 после реального захода — max HP так и остался **427** (это HP для Vigor=11!), а не 1400.

Отсюда различие **CT (безопасно) vs файловый редактор (бан)**:
- CT/тул меняют стат **в памяти живого игрока** → движок сразу пересчитывает HP/FP/Stamina → в сейв
  уходят корректные значения.
- Редактор меняет стат **в файле**, HP/FP/Stamina оставляет старыми, и **игра их сама не чинит** →
  персонаж со статом 99 и производным от 11 → рассинхрон «стат ↔ производное» → серверная проверка → бан.
- **Резисты и хеш `0x60` игра пересчитывает сама** при заходе → редактору о них думать НЕ нужно.

## Что должен чинить редактор (TODO на завтра)

**Проверено по коду редактора (2026-07-12):** `Character.ts` при правке VIG/ATN/END **уже пересчитывает
базовые** HP/FP/Stamina — но пишет только по одному оффсету каждого (`OFFSETS` в `constants.ts`):
HP → `−0x130`, FP → `−0x124`, Stamina → `−0x114`. То есть эксперимент с 10 персонажами
(`make_chars.py` патчил ТОЛЬКО статы+Level) был **жёстче**, чем реальный редактор: в его выводе
базовые HP/FP/Stamina корректны. Не пишутся только **скрытые копии** — и игра их при загрузке
не чинит (подтверждено `diff_derived.py`: after-заход не изменил ни одно из 7 полей):
- **`−0x134` HP-A** — не пишется. Формула проверена на всех 99 сейвах VIG-свипа:
  `HP-A = floor(HPmax × float32(1.3))` (именно float32! int(m*1.3) в double даёт off-by-one
  на vig=50/63/67/92/99). В JS: `Math.floor(hpMax * Math.fround(1.3))`.
- **`−0x128` FP-A** — не пишется; == FPmax на всех 99 сейвах ATN-свипа.
- **`−0x11c` / `−0x118`** копии стамины — не пишутся; все три равны на всех 99 сейвах END-свипа.

**ЭКСПЕРИМЕНТ №2 (make_chars2.py, chars_before2/, тот же день) — гипотеза «копии» ЗАКРЫТА.**
Создали 3 персонажей ровно в состоянии «как после редактора»: стат 99 + Level + корректный
`*_MAX` из таблиц, скрытые копии оставлены старыми. После захода в игру:
- VIG99: HP-A `555 → 1819` = `floor(HPmax × float32(1.3))` — игра пересчитала (перс с угольком);
- ATN99: FP-A `98 → 450` == FPmax — пересчитала;
- END99: Stamina-A/B `95 → 170` == max — пересчитала обе.

**Итоговая модель:** игра при загрузке доверяет из файла статам и `*_MAX`, а вот **копии
(HP-A/FP-A/ST-A/ST-B), резисты и хеш `0x60` пересчитывает сама**. При этом `*_MAX` из статов
она НЕ выводит (эксперимент №1: VIG99 с HP 427 так и остался с 427). Значит:
- Редактор, который пишет стат + правильный `*_MAX` (наш так и делает: `−0x130`/`−0x124`/`−0x114`),
  производит сейв, который после ПЕРВОЙ же оффлайн-загрузки неотличим от игрового в окне персонажа.
  **Дописывать копии в редакторе НЕ нужно.**
- Опасен только редактор, который `*_MAX` не пишет: рассинхрон «стат ↔ max» игра не чинит никогда.

**ЭКСПЕРИМЕНТ №3 (Save Watcher, ивенты no_lvlup / lvlup, 2026-07-12) — флага первого левелапа НЕТ.**
Свежий персонаж (lvl 8), контрольные сейвы без левелапа, затем легитимный левелап 8→11 (VIG+3) у
Хранительницы. Полнослотовый дифф перехода (шум отфильтрован по 14 парам соседних сейвов):
- В окне персонажа изменились ТОЛЬКО ожидаемые поля: VIG, Level, Souls, HP-поля. Стабильных
  «флаговых» байт в ±0x400 от паттерна нет (проверено по всем сейвам обеих групп).
- **Найдено ТРЕТЬЕ HP-поле `−0x138` = current HP** (текущее, не max): после левелапа = 669 =
  embered max; в эксперименте №2 игра его НЕ трогала (осталось 555 при HP-A→1819) — т.е. игра
  доверяет current HP из файла, редактору можно не трогать (меньше max — легально).
- Единственное прочее изменение: таблица 40-байтовых записей `pat+0x46374..+0x46c23`
  (байты `fd`→`00`, float-подобные записи ×3) обнулилась один раз за переход — скорее эффект
  отдыха у костра/варпа (ресет мира), чем левелапа. Проверяется ивентом «rest без левелапа».

**Оставшиеся подозреваемые на бан** (окно персонажа чисто → искать вне его):
1. Сейв уходит на сервер ДО оффлайн-перезагрузки (юзер правит файл и сразу заходит онлайн —
   копии в файле ещё старые? нет: игра чинит их в памяти при загрузке, вопрос в том, ЧТО именно
   уходит в телеметрию до первого ресейва).
2. Soul Memory ↔ Level: редактор меняет Level, не трогая soul memory — классический флаг DS3.
   (Но CT в памяти делает то же самое — надо сравнить, меняет ли игра soul memory при левелапе.)
3. Поля вне окна `[-0x140, +0x10]`: инвентарь, флаги, служебная шапка слота.

## Оставшиеся гипотезы

- HP-A (`−0x134`) — вероятно current HP с бонусом уголька (embered +30% HP). Подтвердить и учесть в редакторе.
- Гипотеза 2 (тайминг античита, офлайн→онлайн) — из байтов не проверяется, поведенческая; правдоподобна.
- Бэкапы сейва перед экспериментами: `DS30000.sl2.bak_20260712_*` в папке сейва.
